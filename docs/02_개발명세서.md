- 데이터 접근 통제는 **RLS로 DB 레벨에서 이중 방어**.

## 5. 인증 · 권한 설계

### 5.1 계정 생성 (관리자 주도)
- 직원 셀프 가입 **없음**. 관리자가 `admin/employees`에서 직원 생성.
- 생성 흐름: 관리자가 이름·이메일·역할 입력 → **서버 액션이 `service_role`로 `auth.admin.createUser()` 호출** → 임시 비밀번호 생성 → `employees` 테이블에 프로필 행 생성 → 임시 비밀번호를 관리자에게 표시(또는 이메일 발송).
- 직원 최초 로그인 시 비밀번호 변경 유도.

### 5.2 역할(Role)
`employees.role`: `employee` | `accountant` | `manager` | `admin`
- `admin` = 대표(최고관리자). `manager` = 부서장(확장용, 1단계 미사용). `accountant` = 회계담당.

### 5.3 권한 매트릭스 (요약)

| 기능 | employee | accountant | admin |
|------|:--:|:--:|:--:|
| 본인 근태/보고/미션 | ✅ | ✅ | ✅ |
| 전 직원 근태/보고 조회 | ❌ | ❌ | ✅ |
| 근태 정정·휴가 승인 | ❌ | ❌ | ✅ |
| 예산 입력 | ❌ | ✅ | ❌(조회만) |
| 예산 조회 | ❌ | ✅ | ✅ |
| 직원·시스템 설정 | ❌ | ❌ | ✅ |
| 게시판/자료/캘린더 | ✅ | ✅ | ✅ |

### 5.4 RLS 정책 원칙
- 모든 테이블 RLS **활성화**.
- 헬퍼 함수로 현재 사용자 역할 조회:
```sql
create or replace function auth.user_role() returns text as $$
  select role from public.employees where id = auth.uid();
$$ language sql stable security definer;
```
- 패턴 예:
  - 본인 데이터: `using (employee_id = auth.uid())`
  - 관리자 전체: `using (auth.user_role() = 'admin')`
  - 예산: `using (auth.user_role() in ('accountant','admin'))`
- 클라이언트가 직접 Supabase에 접근해도 RLS가 차단 → 이중 방어.

## 7. 핵심 기능 구현 상세

### 7.1 출퇴근 위치 인증 (IP + QR) ⭐ 가장 중요

**검증 흐름 (서버측)**
```
출근 버튼 클릭
  → [모바일] 사무실 QR 카메라 스캔 → token 획득
    [PC] QR 하단 인쇄 코드 입력 (또는 웹캠 스캔)
  → 서버 액션 호출 (token 전달)
  → 서버에서 검증:
      ① 요청 IP가 company_settings.allowed_ip_ranges(CIDR) 안에 있나?
         (Vercel: 헤더 'x-forwarded-for'의 첫 IP 사용)
      ② token이 qr_tokens에서 is_active=true 인가?
      ③ 오늘 이미 출근 기록이 있나? (중복 방지)
  → 모두 통과 시: check_in_at = now()(서버시각)으로 기록
  → 지각 판정: now()의 시각 > work_start_time + grace → is_late=true, status='지각'
```

**구현 메모**
- IP 추출은 반드시 서버에서: `request.headers.get('x-forwarded-for')?.split(',')[0]`
- CIDR 매칭 유틸 구현(또는 `ip-cidr` 패키지). 유동 IP 대비 관리자가 `allowed_ip_ranges` 수정 UI 제공(`admin/settings`).
- QR 고정 코드: 사무실에 QR 포스터 부착. QR은 `https://<앱주소>/attendance/qr?token=<code>` 인코딩 → 모바일 스캔 시 자동으로 출근 처리 페이지 진입(로그인 세션 필요).
- 우회 차단 한계 명시: 고정 QR은 사진 공유 가능 → Phase 2에서 TOTP 회전·GPS로 강화 예정.
- 퇴근도 동일 검증(IP). 조기퇴근 판정: `now() < work_end_time`.

### 7.2 근태 통계
- 월별 집계: `attendance`를 `work_date` 기준 집계 → 정상/지각/결근/조기퇴근 카운트.
- 결근 판정: 해당 근무일(평일, 휴가 아님)에 기록 없음 → 배치 또는 조회 시 계산.
- 휴가일은 `leaves`(승인됨)와 대조해 통계에서 제외.
- 화면: 월별 캘린더(색상) + 우측 통계 패널 / 연간 추이(Recharts) / 개인·전체 토글.
- 실시간 출근 현황판: Supabase **Realtime** 구독으로 오늘 `attendance` 변경 실시간 반영.

### 7.3 업무보고 (동적 템플릿)
- `report_templates.fields`(jsonb)를 읽어 **폼을 동적 렌더링**(React Hook Form + Zod 동적 스키마).
- 필수 항목 미입력 시 제출 차단.
- 프로젝트 태그: `projects`에서 선택 → `report_project_links`에 저장(+소요시간).
- 미션 연결: `report_missions`.
- 관리자 화면: 미제출 현황(직원×날짜 매트릭스), 코멘트(`report_comments`), 읽음 표시.
- **자동 취합(Phase 2)**: 주간 보고 작성 시 해당 주의 일일보고를 모아 초안 생성.

### 7.4 미션 & 목표 부합
- 작성 → 승인 → 진행 → 완료 (단순 상태).
- 진행률: 연결된 보고 기반 자동 + 수동 수정 가능.
- 미션 외 업무: 보고가 미션에 연결 안 되면 '기타'로 집계 → 대시보드에 **미션 외 비율** 표시.

### 7.5 연차 (수동 부여)
- 자동계산 **없음**. 관리자가 `employees.annual_leave_total` 직접 입력.
- 휴가 승인 시 `annual_leave_used` 증가, 잔여 = total − used 표시.

### 7.6 알림 (웹 + 이메일)
- 인앱: `notifications` 테이블 + Realtime 구독(헤더 벨 아이콘).
- 이메일: Supabase 이메일 또는 Resend 등 연동(미체크·미제출·승인요청 등 트리거).
- 발송 트리거는 서버 액션/예약 작업(Vercel Cron)으로.

### 7.7 파일 업로드 (Storage)
- Supabase Storage 버킷: `attachments`(증빙·보고첨부), `files`(자료실), `profiles`(프로필).
- 업로드는 서버 경유 또는 서명 URL. DB에는 경로·권한만 저장.
- 버킷별 RLS(Storage policy)로 접근 통제(예산 증빙은 회계·대표만).

### 9.2 예산 🔒 (Phase 2) — 자금 현황 모니터링 *(라이트화)*

> ⚠️ **범위 재정의(중요)**: 기능명세서 5.6의 *매출·매입 손익, 정부과제 분리트랙, 증빙 자동대사, 계정과목별 집행률* 등은 **제외**.
> 실제 구현 범위 = **통장 입출금 내역 · 계정별 잔액 · 법인카드 사용내역의 "현황 조회"** 뿐.
> ("입력은 최소, 보는 건 충실" 철학에 맞춰 회계 ERP 요소를 덜어냄)

**접근 통제**: `accountant` + `admin` 만. RLS는 **SELECT까지** `is_accountant_or_admin()`로 제한(다른 모듈과 달리 *조회도 차단*). 일반 직원은 메뉴 자체 미표시.

**데이터 모델**
```sql
-- 자금 계정 (통장 / 법인카드 / 현금)
create table budget_accounts (
  id uuid primary key default gen_random_uuid(),
  name text not null,                          -- 기업은행 운영통장, 신한 법인카드 ...
  kind text not null default 'bank',           -- bank(통장) / card(법인카드) / cash(현금)
  opening_balance numeric not null default 0,  -- 기초 잔액(시스템 시작 시점 1회 입력)
  is_active boolean not null default true,
  sort int not null default 0,
  note text,
  created_at timestamptz not null default now()
);

-- 거래 내역 (입금/출금/카드결제 통합 원장)
create table budget_transactions (
  id uuid primary key default gen_random_uuid(),
  txn_date date not null,
  direction text not null,                     -- in(입금) / out(출금·카드결제)
  amount numeric not null default 0,           -- 양수, 부호는 direction이 결정
  summary text,                                -- 적요(거래 내용)
  counterparty text,                           -- 거래처(선택)
  account_id uuid references budget_accounts(id) on delete set null,
  note text,
  created_by uuid references employees(id),
  created_at timestamptz not null default now()
  -- ※ category·gov_project_id 컬럼은 향후 확장 여지로 보유 가능하나, 라이트 버전 UI에선 미사용
);
```
RLS: 두 테이블 모두 `for all to authenticated using (public.is_accountant_or_admin()) with check (public.is_accountant_or_admin())`.

**핵심 로직**
- **계정 잔액 = `opening_balance + Σ(in) − Σ(out)`** (해당 account_id 거래 합산). 통장은 입·출금, 카드는 결제(out)만 발생.
- **통장 입출금 현황** = `kind='bank'` 계정 거래. **법인카드 사용내역** = `kind='card'` 계정 거래(전부 out).
- **카드 사용 ≠ 통장 출금**: 카드 계정 거래(카드 사용)는 ‘출금’ 합계·순증감에서 **제외**(`txnBucket()`로 in/out/card 3분류). 실제 카드대금은 결제일에 통장 ‘출금’으로 별도 입력 → **이중 계상 방지**. (카드 계정은 ‘잔액’ 대신 ‘이달 사용’ 표기)
- 도입 시 회계담당이 각 통장·카드의 **기초잔액을 1회 등록** → 이후 거래만 입력하면 잔액이 자동 추정됨.

**화면**
- `/budget` — **일별 현황**(회계담당이 매일 확인·입력): 날짜 내비게이터(◀ 오늘 ▶ + 달력)로 **그날의 입금/출금/카드 사용 합계 + 거래 목록**을 보고 입력, + 통장 잔액 카드 + **월 일별 요약 표**(날짜별 입금/출금/카드/순증감·건수, 행 클릭 시 해당일로 이동). **적요는 필수 입력**.
- `/budget/accounts` — 자금 계정 관리(통장·카드 등록, 기초잔액, 활성/비활성).
- `/budget/stats` — **월별·연도별 통계**: 입금/출금/카드 사용·순증감을 표 + 막대그래프(Recharts)로. 연도 선택, 카드는 출금과 분리 집계.
- (선택) 대표 현황판에 "오늘 입력 여부"는 *당일 거래 존재*로 단순 판단(별도 마감 테이블 없음).

**구현 메모**
- `lib/auth.ts`에 `getBudgetOrThrow()`(accountant/admin) 추가 — 서버 액션 진입 가드. 위반 시 throw.
- 쓰기는 역할 확인 후 `createAdminClient()`(또는 RLS 통과 user client). 금액은 `numeric`, 표기는 `₩#,###`(`lib/budget.ts` 포맷터).
- 메뉴 노출: `components/dashboard/nav.ts`의 예산 항목 `roles:['accountant','admin']`, `phase:1`(활성)로 변경. `/budget/layout.tsx`에서 `requireRole(['accountant','admin'])` 게이트.

**제외(라이트화)**: 매출/매입 손익·부가세, 수금/미수 추적, 정부과제 비목 관리, 증빙 자동대사, 예산편성 대비 집행률, 세무자료 내보내기. (필요 시 별도 검토)

### 9.3 게시판 (Phase 3) — 공지 + 자유 통합

기능명세서 5.7 대응. 공지·자유·익명건의를 **카테고리로 구분**한 단일 게시판.

**데이터 모델**
```sql
create table board_posts (
  id uuid primary key default gen_random_uuid(),
  category text not null default '자유',     -- 공지 / 자유 / 건의(익명)
  title text not null,
  body text,                                  -- 리치 HTML (DOMPurify 정화 후 저장/렌더)
  author_id uuid not null references employees(id),
  is_pinned   boolean not null default false, -- 상단 고정
  is_required boolean not null default false, -- 필독(공지)
  is_popup    boolean not null default false, -- 로그인 시 팝업 표시(공지)
  publish_at  timestamptz,                    -- 예약 발행(미래면 미게시)
  view_count  int not null default 0,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create table board_comments (                 -- 댓글 + 대댓글
  id uuid primary key default gen_random_uuid(),
  post_id uuid not null references board_posts(id) on delete cascade,
  author_id uuid not null references employees(id),
  parent_comment_id uuid references board_comments(id) on delete cascade,
  content text not null,
  created_at timestamptz not null default now()
);

create table board_reads (                    -- 공지 읽음 확인(수신 명단)
  post_id uuid references board_posts(id) on delete cascade,
  employee_id uuid references employees(id) on delete cascade,
  read_at timestamptz not null default now(),
  primary key (post_id, employee_id)
);

create table board_reactions (                -- 이모지 반응
  post_id uuid references board_posts(id) on delete cascade,
  employee_id uuid references employees(id) on delete cascade,
  emoji text not null,
  primary key (post_id, employee_id, emoji)
);

create table board_popup_dismissals (         -- 공지 팝업 "다시 보지 않기"
  post_id uuid references board_posts(id) on delete cascade,
  employee_id uuid references employees(id) on delete cascade,
  scope text not null,                        -- today(오늘 하루) / forever(다시 안봄)
  created_at timestamptz not null default now(),
  primary key (post_id, employee_id)
);
```
**RLS**
- `board_posts`: SELECT 전 직원 — **단 `category='건의'`(익명)는 작성자 본인 + admin만** 조회. INSERT는 본인(author=auth.uid()), `category='공지'`는 앱 레벨에서 admin만 허용. UPDATE/DELETE는 작성자 또는 admin.
- comments/reactions/reads: SELECT 인증 직원, INSERT 본인.

**핵심 로직 — 로그인 팝업 공지**
1. 로그인 직후 `is_popup=true AND (publish_at is null or publish_at<=now()) AND category='공지'` 공지 조회.
2. 그중 사용자의 `board_popup_dismissals(scope='forever')` 또는 **오늘자 `scope='today'`** 가 *없는* 공지만 모달로 표시.
3. 모달의 "오늘 하루 안보기/다시 안보기" → dismissals 기록.

**화면**: `/board` 목록(공지 상단고정·필독 뱃지), `/board/[id]` 상세+댓글·이모지, 작성(리치에디터·첨부 `uploads`), 관리자 읽음현황. `@멘션`은 본문/댓글 파싱 → `notifications` 발송.

**제외(라이트화)**: 신고·블라인드 자동화, 정교한 권한그룹.

### 9.4 자료공유 (Phase 3) — 공용 자료실

기능명세서 5.8 대응. 사규·양식·매뉴얼 등 보관. **업로드형 + 외부링크형 병행**.

**데이터 모델**
```sql
create table file_folders (
  id uuid primary key default gen_random_uuid(),
  name text not null,
  parent_id uuid references file_folders(id) on delete cascade,  -- 폴더 트리
  sort int not null default 0,
  created_at timestamptz not null default now()
);

create table shared_files (
  id uuid primary key default gen_random_uuid(),
  folder_id uuid references file_folders(id) on delete set null,
  name text not null,
  storage_path text,                       -- Supabase Storage 경로(업로드형)
  external_url text,                       -- 구글드라이브 등 외부 링크(대용량형)
  mime_type text,
  size_bytes bigint,
  visibility text not null default 'all',  -- all(전체) / restricted(특정 인원)
  uploaded_by uuid references employees(id),
  created_at timestamptz not null default now()
);

create table file_grants (                 -- visibility='restricted' 시 열람 허용 대상
  file_id uuid references shared_files(id) on delete cascade,
  employee_id uuid references employees(id) on delete cascade,
  primary key (file_id, employee_id)
);
```
- **Storage 버킷 `files`(비공개)** — 다운로드는 서버에서 **서명 URL** 발급. 예산 증빙은 여기 노출 안 함(예산 모듈 내에서만).
- **RLS**: `shared_files` SELECT = `visibility='all' OR uploaded_by=auth.uid() OR is_admin() OR (file_grants에 본인 존재)`. 쓰기는 업로더/admin. `external_url`은 대용량 파일을 Storage 용량 없이 공유.

**화면**: `/files` 폴더 트리 + 파일 목록, 업로드(파일 또는 외부링크 등록), 미리보기(이미지·PDF), 검색·정렬. **버전 = 최신본 덮어쓰기**(이력 미관리 — 라이트).

**제외(라이트화)**: 버전 히스토리, 풀 권한그룹, 동시편집.

### 9.5 캘린더 (Phase 3) — 통합 일정 뷰

기능명세서 5.9 대응. **각 모듈에 흩어진 날짜를 모아 보는 뷰**가 핵심(풀 일정관리 아님).

**데이터 모델** — 직접 등록 이벤트만 저장, 나머지는 *조회 시 합성*
```sql
create table calendar_events (
  id uuid primary key default gen_random_uuid(),
  title text not null,
  event_type text not null default 'personal', -- personal/company/project/meeting
  start_date date not null,
  end_date date,
  all_day boolean not null default true,
  start_time time, end_time time,               -- all_day=false 시
  owner_id uuid references employees(id),
  visibility text not null default 'shared',    -- private(개인) / shared(공유)
  note text,
  created_at timestamptz not null default now()
);
```
**자동 수집(읽기 전용, 별도 저장 없음)** — 서버에서 다음을 UNION하여 표기:
- 휴가: `leaves`(status='승인') → 부재 일정
- 프로젝트: `project_milestones.date`, `projects.end_date`(기한)
- 미션: `missions.period_start ~ period_end`
- (선택) 공지 행사: 일정형 `board_posts`
- 공휴일: **`date-holidays`**(기 도입) — 원본 수정은 각 모듈에서, 캘린더는 표시만.

**RLS**: `calendar_events` SELECT = `visibility='shared' OR owner_id=auth.uid() OR is_admin()`. 쓰기는 owner/admin.
**화면**: `/calendar` 월/주/리스트 뷰, **부재 현황**(누가 언제 휴가), 유형별 색상, 공휴일 표시.

**제외(라이트화)**: 회의실 예약, 반복일정(RRULE), 외부 캘린더 동기화.

### 9.6 전자결재 (Phase 3) — 간단 신청 → 승인

기능명세서 5.10 대응. **결재선·전결·대결 엔진 없이** 단일 신청→승인.

**데이터 모델**
```sql
create table approvals (
  id uuid primary key default gen_random_uuid(),
  form_type text not null,             -- leave(휴가)/expense(지출)/purchase(구매)/general(기안)
  title text not null,
  content jsonb not null default '{}',  -- 양식별 입력 필드값
  applicant_id uuid not null references employees(id),
  approver_id uuid references employees(id),     -- 보통 대표
  status text not null default '대기',  -- 대기 / 승인 / 반려 / 회수
  reject_reason text,                   -- 반려 시 필수
  attachment_path text,                 -- 증빙(Storage)
  linked_leave_id uuid references leaves(id),     -- 휴가결재 승인 시 생성/연결
  decided_by uuid references employees(id),
  decided_at timestamptz,
  created_at timestamptz not null default now()
);

create table approval_comments (
  id uuid primary key default gen_random_uuid(),
  approval_id uuid references approvals(id) on delete cascade,
  author_id uuid references employees(id),
  content text not null,
  created_at timestamptz not null default now()
);
```
**흐름**: 신청(대기) → 대표 **승인/반려**(반려사유 필수) / 신청자 **회수**(상태='대기'일 때만). 신청·결과는 `notifications`(+이메일) 발송. 대표 **모바일 승인함** = `status='대기'` 목록.
**연동**: `form_type='leave'` 승인 시 → `leaves`에 승인건 반영 → 근태 통계·캘린더 자동 표기. *(기존 근태의 휴가신청과 통합할지 / 휴가를 전자결재로 일원화할지는 구현 착수 시 택1)*
**RLS**: SELECT = 본인(applicant) 또는 approver/admin. INSERT applicant=auth.uid(). UPDATE(결재) approver/admin, 회수는 본인 & status='대기'.
**화면**: `/approvals` — 내 신청 / 받은 결재 탭, 신청 폼(양식별 동적), **상세 = 정식 결재문서 양식**(결재란·도장칸·문서번호·회사명) + 코멘트, 대표 승인함.
**출력**: 브라우저 인쇄(`window.print()` + `globals.css @media print`로 `.print-document` 영역만 출력) → PDF 저장. 문서는 **A4 세로(210×297mm) 고정**, 하단에 **회사 직인**(`company_settings.seal_image_url`)·승인란에 **결재자 서명**(`employees.signature_image_url`) 이미지 표시(관리자 설정 화면에서 업로드, 투명 PNG 권장). ⚠️ **hwpx 직접 생성은 미지원** — 신뢰할 JS 생성 라이브러리 부재 + 서버리스(한컴 자동화 불가). 편집 가능한 office 파일이 필요하면 **DOCX 내보내기**(한글에서 열림)로 후속 대응 가능.

**제외(라이트화)**: 다단계 결재선, 전결/대결 규정, 금액별 자동 결재선.
