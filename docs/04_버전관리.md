## 1. 버전 체계 (Semantic Versioning)

`MAJOR.MINOR.PATCH` 형식을 따른다.

| 자리 | 올리는 경우 | 예시 |
|------|------------|------|
| **MAJOR** | 호환되지 않는 큰 변경, 대규모 개편 | 1.0.0 → 2.0.0 |
| **MINOR** | 하위 호환되는 기능 추가(새 모듈 등) | 1.0.0 → 1.1.0 |
| **PATCH** | 버그 수정·소규모 개선 | 1.0.0 → 1.0.1 |

- 모든 버전 변경은 [`CHANGELOG.md`](../CHANGELOG.md)에 기록한다.
- 릴리스 시 Git 태그 `vX.Y.Z`를 부여한다.

## 3. 브랜치 전략

소규모 팀 기준 단순 전략:

- **`main`** — 항상 배포 가능한 상태. Vercel이 이 브랜치를 자동 배포.
- **`feature/*`** — 기능 개발 브랜치. 완료 후 `main`으로 병합(PR 권장).
- 핫픽스는 `fix/*` 후 `main` 병합 → PATCH 버전 증가.

> 직접 `main` 커밋은 지양하고, 변경은 브랜치 → 병합으로 추적한다.

## 4. DB 마이그레이션 버전 관리

`supabase/migrations/` 에 번호 순으로 SQL을 누적한다. **재실행 안전(idempotent)** 하게 작성.

| 파일 | 내용 |
|------|------|
| `0001_init.sql` | 인사·근태·업무보고·미션·프로젝트·알림 + RLS |
| `0002_budget.sql` | 예산(자금계정·거래원장, 라이트) |
| `0003_phase3.sql` | 게시판·자료공유·캘린더·전자결재 |
| `0004_sales_targets.sql` | 매출 목표 + 거래 category 복원 |
| `0005_backups.sql` | 백업 메타테이블 |

- 스키마 변경 시 **새 번호 마이그레이션 파일 추가**(기존 파일 수정 지양).
- 적용은 Supabase SQL Editor 또는 Management API.

## 5. 릴리스 절차

1. 기능 개발·검증(타입체크·린트 통과) → `main` 병합.
2. `CHANGELOG.md` 갱신, 버전 결정(SemVer).
3. `package.json`의 `version` 갱신.
4. `git tag vX.Y.Z` → push.
5. Vercel 자동 배포 확인, 환경변수·마이그레이션 반영 여부 점검.

## 6. 환경/배포 체크리스트

- [ ] Vercel 환경변수 5종 등록(`NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `QR_KIOSK_SECRET`, `CRON_SECRET`).
- [ ] DB 마이그레이션 `0001~0005` 적용 확인.
- [ ] Storage 버킷 `uploads`·`files`(공개), `backups`(비공개) 생성 확인.
- [ ] 관리자 설정에서 **허용 IP를 회사 공인 IP**로 교체(임시값 `192.168.101.0/24`).
- [ ] 자동 백업 Cron(`/api/cron/backup`) 동작 확인.
